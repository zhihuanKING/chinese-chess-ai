## 摘要

本文在 **8×A800(40GB)单机、约一天训练预算**的强约束下,设计并实现了一个中国象棋学习型博弈 AI。核心方法为**两阶段管线**:先用 Pikafish 自对弈生成的约 230 万局面做**监督冷启动**,再以 **AlphaZero 范式 + 向量化批量 MCTS** 进行自对弈强化学习,并以传统 **Alpha-Beta 引擎**作对照基线。系统采用 **C++ 规则内核(pybind11 暴露)+ Python 训练/搜索**的混合架构,规则内核以 perft 对拍达到逐位正确(perft(1..5)=44/1920/79666/3290240/133312995)。实验表明 [待填:Elo 上升曲线 / 反超 Alpha-Beta 的 crossover / Gumbel 低模拟样本效率]。本文的贡献在于:在有限算力下量化刻画了样本效率、模拟预算与最终棋力之间的权衡关系。

**关键词**:中国象棋;AlphaZero;Gumbel-AlphaZero;蒙特卡洛树搜索;监督冷启动;Alpha-Beta 搜索

## 第1章 绪论

### 1.1 研究背景
博弈一直是人工智能的试金石。从深蓝(Deep Blue)以 Alpha-Beta 搜索+人工评估战胜国际象棋世界冠军,到 AlphaGo(2016)、AlphaGo Zero(2017)、AlphaZero(2018)以"深度神经网络 + 蒙特卡洛树搜索(MCTS)+ 自对弈强化学习"统一了完全信息博弈的求解范式,博弈 AI 经历了从"人写规则"到"自我学习"的代际跨越。2022 年 Gumbel-AlphaZero 进一步在低模拟预算下提供了带理论保证的策略改进,2024–2025 年又涌现 ReZero、EfficientZero V2、Search-contempt 等面向有限算力的改进。

### 1.2 中国象棋的博弈学定性
中国象棋是**完全信息、确定性、双人零和、回合制**博弈:无隐藏信息、无机会节点。这一定性决定了算法选型——它属于 AlphaZero/Alpha-Beta 擅长的领域,而不适用于面向不完全信息的 CFR 系方法(见第 2 章)。

### 1.3 研究问题与创新点
**研究问题**:在仅 8×A800、约一天训练预算的强约束下,"监督冷启动 + 低模拟自对弈"两阶段管线能否训练出显著超越同等时间预算 Alpha-Beta 引擎的学习型 AI?

**创新点**:
1. 以 **Gumbel-AlphaZero(ICLR 2022)** 作为核心训练算法,低模拟数仍保证策略改进;
2. **Pikafish 自对弈监督冷启动 + RL 微调**两阶段管线,破解一周/一天算力约束(并规避了人类棋谱下载受限的工程问题);
3. **C++ 规则内核 + Python 向量化批量 MCTS** 的混合架构,以集群级聚合利用率吃满 8 卡;
4. 针对象棋的定制:from→to 动作编码 + 合法掩码 + 红黑对称视角规范化。

第 2 章相关技术调研;第 3 章环境建模与正确性验证;第 4 章算法设计;第 5 章工程实现与系统架构;第 6 章实验设置;第 7 章实验结果与分析;第 8 章总结与展望;附录含超参、复现说明、参考文献与开源声明。

## 第2章 相关技术(调研)

按"完全信息 vs 不完全信息"分水岭组织,覆盖任务书点名的四类技术。

### 2.1 Minimax 与 Alpha-Beta 剪枝
经典对抗搜索。本文基线引擎实现了 negamax + α-β 剪枝 + 置换表(Zobrist)+ 迭代加深 + 静态搜索(quiescence)+ MVV-LVA 走子排序,配分阶段手写评估。

### 2.2 MCTS / PUCT
蒙特卡洛树搜索以"选择-扩展-评估-回传"四步增量构建搜索树;AlphaZero 采用 PUCT 选择 `a*=argmax[Q+c_puct·P·√ΣN/(1+N)]`,以神经网络先验 P 引导搜索。

### 2.3 AlphaGo Zero / AlphaZero
策略-价值双头网络 + MCTS 自我对弈,无需人类知识即可从零超越顶级程序;在将棋(与象棋同构)上 <24h 达到超人水平。

### 2.4 Gumbel-AlphaZero(核心)
Danihelka et al., ICLR 2022。根节点以 Gumbel-Top-k + Sequential Halving 在少量模拟内集中预算,非根节点用 completed-Q 确定性选择,**在低模拟数(16~32)下仍有策略改进的理论保证**,样本效率较原版高约一个数量级。这是本文"有限算力破局"的算法答案。

### 2.5 MuZero(辨析:不实现)
Schrittwieser et al., 2020。价值在于环境模型未知时联合学习 dynamics;中国象棋规则完全已知,学 dynamics 徒增三网联合与调参复杂度且无精度优势,故仅调研、不作主线。

### 2.6 不完全信息博弈方法(辨析:不适用)
DeepCFR / DeepStack / Libratus / Pluribus / Suphx 针对扑克、麻将等**不完全信息**博弈,在 information set 上求纳什均衡;象棋无隐藏信息,CFR 框架无作用对象。Student of Games(Science 2023)、AlphaZe**(2023)则尝试统一两类博弈,可作前沿展望。

### 2.7 前沿动态(2024–2026)
ReZero(reanalyze 加速)、EfficientZero V2、UniZero、Search-contempt(有限算力训练)、Searchless Chess / Chessformer / Maia(Transformer 路线)。本文以 Gumbel-AZ 为核心并借鉴 KataGo 的工程省样本技巧(playout cap randomization 等)。

## 第3章 环境建模与正确性验证

### 3.1 棋盘与走法
9 列 × 10 行 = 90 格,mailbox(int8[90])表示。实现完整走法:将/帅(九宫内、飞将判定)、士(九宫斜走)、象(田字、塞象眼、不过河)、马(日字、蹩马腿)、车(直线滑动)、炮(隔且仅隔一子吃)、兵卒(过河前只进、过河后可平不可退)。合法着法过滤送将/自杀着。

### 3.2 规则与终局
将军/应将、将死、困毙(无合法着且未被将判负)、第 3 次重复判和、120 回合(240 半回合)自然限着判和。长将/长捉本版按重复处理(简化版,报告声明,完整规则列展望)。

### 3.3 表示与编码
Zobrist 增量哈希(含行动方位,make/unmake 对称)、FEN 往返无损;动作编码 **move = from×90 + to ∈ [0,8100)**,配合合法掩码(2086 紧凑编码列为后续消融);输入特征 C×10×9(C=15:己方7+对方7+重复计数1),**以当前行动方视角规范化**(黑走时上下翻转+红黑互换)。

### 3.4 正确性验证(最高优先级)
- **perft 对拍**:从初始局面 perft(1..5) = **44 / 1920 / 79666 / 3290240 / 133312995**,与公开参考值逐位一致。
- **规则单元测试**:飞将、蹩马腿、塞象眼、炮隔子吃、送将过滤等共 31 项断言全部通过。
- **一致性**:Zobrist make/unmake 对称(200 步压力测试增量哈希=重算)、FEN 往返无损、输入编码↔解码视角一致(红黑对称专项测试通过)。

## 第4章 算法设计

### 4.1 策略-价值网络
ResNet 主干:Conv3×3(C)+BN+ReLU → N 个残差块 → 双头。Policy 头:Conv1×1→Flatten→Linear→8100 logits(+合法掩码,非法置 -inf 后 softmax);Value 头:Conv1×1→Flatten→Linear→Linear(1)→tanh∈[-1,1]。本文用 **128 通道×10 块(≈26.4M 参数)**作主力,256×15(≈41.3M)作规模消融。

### 4.2 损失函数
`L = (z−v)² − πᵀlog p + c‖θ‖²`,c=1e-4(L2 由 AdamW weight_decay 实现)。Gumbel 版 π 用 completed-Q 改进策略作目标。

### 4.3 搜索:PUCT 与 Gumbel-MCTS
统一 `Planner.search(positions, net, n_sim)` 接口,**向量化批量**实现:同时处理 B 局并行对局,把各局待评估叶子拼成一个大 batch 一次喂 GPU(吃满 GPU 的关键)。PUCTPlanner(根节点 Dirichlet 噪声、虚拟损失)先行跑通学习曲线;GumbelPlanner(Sequential Halving + completed-Q)作核心创新对照。

### 4.4 两阶段训练管线
1. **监督冷启动**:用 Pikafish 自对弈生成的 (局面, 着法, 结果) 监督训练 policy+value,快速跳过随机乱走早期;
2. **自对弈 RL**:actor-learner 架构,自对弈样本入共享 ReplayBuffer,learner 采样训练并周期导出权重,worker 热加载;Elo-gating 晋升最强权重。

### 4.5 Alpha-Beta 基线
见 2.1;接 UCI 子集便于 Arena 对弈与同时间预算对照。

## 第5章 工程实现与系统架构

### 5.1 混合架构与语言边界
- **C++17**:规则内核(perft 硬验证)+ Alpha-Beta 引擎,经 **pybind11** 暴露为 `xqai._xqcore`,经 scikit-build-core 随包编译;
- **Python + PyTorch 2.6(cu124)**:网络、向量化 MCTS、自对弈、训练、评估;
- 唯一跨语言契约 = checkpoint 权重文件,进程级解耦=风险隔离。

### 5.2 向量化批量 MCTS(吞吐杠杆)
单进程维护 B 局并行对局,叶子拼 batch 一次推理(参考 TurboZero/MCTX)。**权衡说明**:相比 C++ 多进程集中推理服务,本文选择 Python 向量化 + 多 worker 进程,以最小化系统复杂度换取一天约束下的迭代速度与正确性确定性,用"每卡多 worker + 集群聚合利用率"补回单任务效率。

### 5.3 分布式拓扑(独享 8×A800)
GPU0 learner + GPU1–7 自对弈(每卡多 worker);ReplayBuffer 驻 /dev/shm(多进程共享内存,容量 200 万局面);数据/权重落 NVMe。worker 轮询 latest.pt 的 mtime 热加载。

### 5.4 工程实现中的关键问题与解决(体现工程深度)
- **冷启动加载**:learner 按 checkpoint 内记录的 channels/blocks 建网,加载失败即显式中止,杜绝"静默退化为从零训练"浪费算力;
- **自对弈喂数据延迟**:原实现每局全部走完才 flush 样本导致 learner 饿死,改为分块 flush 使样本秒级入 buffer;
- **多进程清理**:SIGTERM/超时下保证 worker 先停、再清共享内存,无僵尸进程、无 /dev/shm 泄漏;
- **混合精度**:BF16(A800 原生)。

### 5.5 自动化与无人值守
端到端流水线脚本(setsid 脱离 SSH)串联:数据生成→监督预训练→RL→Elo 评估,并配健康巡检(loss/学习步/GPU 利用率/进程存活)自动发现并修正异常。

## 第6章 实验设置

### 6.1 硬件与软件
8× NVIDIA A800-SXM4-40GB(NVLink)、2× AMD EPYC 7K62(192 线程)、1TB 内存;CUDA 12.4 / 驱动 550 / PyTorch 2.6.0+cu124 / g++ 11.4。

### 6.2 数据
监督冷启动数据由 Pikafish(NNUE,约 58 万 NPS)自对弈生成:随机开局 k∈[2,8] 步增多样性,depth 8 搜索,**约 1.2 万局 / 230 万局面 / 572 分片**;红/黑/和 ≈ 33.8%/23.9%/42.3%。

### 6.3 棋力评估方法
Arena 配对开局、红黑各半;NetPlayer(net+MCTS)对 Alpha-Beta(xqab,固定 200ms)对弈,胜率经 Wilson 95% CI 与 logistic 公式反推 Elo;每新 checkpoint 自动评估,构建 Elo-训练步曲线。Pikafish 限档可作绝对棋力锚点。

### 6.4 评估协议
消融统一单变量、配对开局;主曲线每 10 分钟一个评估点(20 局);关键结论可加大局数收紧 CI。
