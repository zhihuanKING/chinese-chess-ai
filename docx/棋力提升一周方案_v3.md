# 棋力提升一周方案 v3.1（事件驱动流水线版）

> 2026-06-11 制定。v3.1 取消按天排班，改为**依赖图 + 事件触发**：每个任务的启动条件是前置产物就绪，而不是日历；GPU/CPU/代码三条泳道全周不空转。
> 前提结论（已实验证实）：监督蒸馏是绝对棋力唯一已证来源；RL 管线机制正确（frzV −301→+135）；net-vs-net Elo 严重通胀（+301 内部 ≈ 0 外部）；瓶颈 = value 头质量 + 训练分布偏移 + 单步低模拟迭代。

## 0. 目标与唯一度量

- **唯一进度指标：外部锚点 Elo 阶梯**（random / 自研AB 200・500ms / Pikafish d1–d4），≥64 局/点、修复版协议（带着法历史）。每次冠军更替/训练轮结束即测（C 完成后单次 ~20min），不是每晚一次。
- **一周目标**：必达 = 锚点平均 +100 Elo、vs AB-500ms ≥ 35%；期望 = vs AB-500ms > 50%、vs pf_d1 ≥ 35%；冲刺 = pf_d1 翻正。

## 1. 关键提速发现（v3.1 相对 v3 的结构性变化）

1. **旧标签离线可修，零引擎成本**：v2 shard 的 `z = tanh(cp/500)` 可逆（`cp = 500·atanh(z)`，fp16 精度在中段足够、饱和区 WDL 同样饱和无损失）；`pi = softmax(cp/100)` 的温度重调 = `pi^(100/T)` 重归一化。**→ 新增 A0：对现有 5.32M 数据做纯 CPU 标签变换 + 立即重训**，开工当天就出第一轮棋力验证，不等新标注。
2. **D2 基线评测的对局本身就是数据**：3 底座 × 7 锚点 × 64 局 ≈ 1300+ 盘对引擎局，全部局面收割进标注队列（败局优先）——测量顺带产出最高价值的 value 盲区标本。
3. **标注流水化**：自对弈 shard 一落盘，CPU 标注 worker 即消费；训练不等"全部标完"，攒够 ~150–200 万就开第一轮，后续数据滚动并入。
4. **A 与 B 统一进一个门控**：蒸馏轮产物和 RL learner 的候选**进同一个 gate**，谁赢谁晋升冠军。不再有"A 阶段/B 阶段"的硬切换——一条冠军线，两个候选生成器（连续 RL + 周期性蒸馏大轮）。
5. **B 提前 ~2 天启动**：关键路径从 "D→A 全量标注→A3→B（第 4 天）" 压缩为 "D1→A0/round-1→B（T+24~30h）"，复利循环从 3 天变 **5.5 天**。

## 2. 依赖图与事件触发

```
T0 ─┬─ [代码泳道] D1 锚点协议修复(2h) ──→ M1: D2 三底座基线(4卡,~3h) ──┐
    ├─ [代码泳道] A0 标签变换脚本(2h) ─→ M2: A0 三变体重训(2-4卡) ────┤
    ├─ [代码泳道] C arena向量化(agent并行写,~半天+验证) ─────────────┤
    ├─ [GPU泳道]  A1 自对弈采集即刻启动(4卡,当前最强网,不等选型*) ────┤
    └─ [CPU泳道]  标注服务常驻启动(150-180线程,优先级队列,全周不停) ──┤
                                                                      ↓
M3: A0 变体锚点对比 → 定标签配方(value公式+温度T) ──→ (若全平: 触发§5.R1)
M4: 新标注 ≥150万 → round-1 全量训练(8卡DDP,~3h,自对弈暂停让位) → 锚点
M5: round-1 冠军就绪(预计 T+24~30h) → B 门控循环启动(6自对弈+1learner+1gate/ladder)
     └─ 飞轮: CPU泳道转为持续标注 B 的新鲜自对弈局面,每~12h并入rehearsal池
M6: 周期性(每~36h)蒸馏大轮 → 作为候选进 gate 与 RL 候选同台竞争
M7: T+7天−6h 冻结 → 最强模型 128局/锚点终测 + 复盘
```
\* A1 不等 D2 选型的依据：DAgger 要的是"我们这族策略的实战分布"，任一候选底座的对局局面都是有效训练数据，混合分布反而更鲁棒；底座选型只影响**重训的初始化**，不影响数据。

## 3. 三条泳道的常驻规则

### GPU 泳道（8×A800，优先级：训练窗口 > 自对弈 > 评测）
- **自对弈是默认负载**：任何卡空下来就归自对弈（采集期 4–6 卡，B 期 6 卡）。
- **训练窗口抢占制**：DDP 全量训练（~3h）直接暂停自对弈 worker，训完恢复（worker 本就轮询冠军权重，暂停无状态损失）。A0 小变体训练用 2 卡/个，与 D2 错峰。
- **1 卡常驻评测**：gate 对局 + 锚点 ladder；C 完成前用 4 分片并行顶替。
- liauto 共存：争抢对称可接受；8 卡 DDP 窗口尽量挑空隙。

### CPU 泳道（192 线程，标注服务全周 24×7 不停）
- 常驻 150–180 线程 Pikafish depth14 MultiPV4；预留 ~30 线程给引擎对手与系统。
- **优先级队列**：① 对引擎败局局面（value 盲区标本）② 最新自对弈局面 ③ 历史回填。
- 标签即产即用：落盘为 sparse shard，训练侧滚动装载（flat memmap 管线，禁稠密 npz 直读——OOM 教训）。
- worker 带自动重启（引擎偶发崩溃已观测 2 次）。
- 吞吐（v2 实测）：~15 万局面/h → 每 24h 产 ~360 万标注。

### 代码泳道（我 + 并行 agent）
- T0 三件事并行发包：D1 协议修复、A0 变换脚本、C 向量化（各一个 agent，对抗审查后合入——项目惯例）。
- 之后转值守：盯 ladder、处理崩溃、按 §5 决策规则切换资源。

## 4. 各任务技术要点

- **D1**：引擎封装补 `position fen <开局> moves <历史>`；锚点全部重测（旧 30 局基线 ±200 Elo 噪声太大，且重复盲区利好被动棋风）。
- **D2**：底座三选一（v1 final_best / v2 冷启动 / safe）。依据：旧观察"v2 监督低搜索弱于 v1"，必须实测定夺。对局全收割。
- **A0**：三个标签变体小步快跑——(a) 仅 value 改 WDL 校准；(b) value 改 + T=50；(c) value 改 + T=30 加尖。WDL 校准用自家数据自标定：旧 raw 对局可复盘出终局结果，对 (恢复的 cp, 实际胜负) 拟合 logistic —— 比照搬 Stockfish 参数更贴本域。变体各训 ~2h（2 卡），锚点 32 局快筛。
- **A1/round-1**：自对弈 n_sim=128、温度多样化；新数据:旧重标数据 ≈ 70:30；8 卡 DDP。
- **C**：arena 复用 `PUCTPlanner.search(positions)` 的批量接口做 N 局 lockstep；与串行版同种子一致性抽查后上线。收益：ladder 1.5h→20min、gate 提速、实战 n_sim 200→800+ 可行（n_sim 60→600 胜率翻倍已证）。
- **B**：Gumbel n_sim 32→128（m=16）+ playout-cap 随机化（~25% 着 200sim 出 policy 目标，其余 64sim 只出 value）；gate ≥0.55（60 局）晋升、冠军实时喂自对弈；rehearsal 25% 引擎标注数据（safe 臂证明无害到有益）；**蒸馏大轮与 RL 候选同 gate 竞争**。

## 5. 决策规则（事件触发，无固定检查日）

- **R1 — A0 三变体锚点全平**：value 假说减弱 → 资源转向：(i) 新鲜引擎标注照常推进 round-1（分布修复仍可能独立起效）；(ii) 并行排一个**大网候选**（320×20 或 256×20，同数据重训）进 gate——容量是已证可靠的杠杆。
- **R2 — B 连续 24h 无晋升**：先注入蒸馏大轮候选；再停滞则 gate 门槛 0.55→0.52 或换 SPRT；仍停滞执行 R1(ii) 大网杠杆。
- **R3 — 标注吞吐 < 计划 60%**：MultiPV 4→2（+60% 吞吐）；30% 局面 value-only depth12；败局局面永远优先。
- **R4 — 任何环节锚点回退 >50 Elo**：冠军回滚（gate 本就保留 best.pt 历史），事故局面入标注队列。

## 7. 时间轴（预期，事件可前后浮动）

- **T+0~5h**：D1/A0/C 三路代码齐发；A1 采集 4 卡起跑；CPU 标注服务起；D2 基线 4 卡跑完 → 底座定。
- **T+5~12h**：A0 三变体训完快筛 → **第一份棋力信号**；C 上线；A1 继续。
- **T+24~30h**：round-1 全量训完 + ladder → **B 启动**，飞轮转起来。
- **T+30h ~ T+6.7d**：B 复利循环 ≈ **5.5 天**，期间 2–3 次蒸馏大轮进 gate，ladder 连续记录。
- **T+6.7d~7d**：冻结、128 局/锚点终测、复盘归档（`logs/ladder/`，Elo-vs-时间主图用外部锚点坐标）。

## 8. 风险登记（对策见 §5）
