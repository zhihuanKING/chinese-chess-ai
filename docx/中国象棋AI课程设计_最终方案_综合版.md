## 2. 算法路线权衡（综合算法研究员视角）

**问题定性决定一切**：中国象棋是**完全信息 + 确定性**博弈 ⇒ 无隐藏信息、无机会节点。

| 算法 | 角色定位 | 理由（论文依据） |
|---|---|---|
| **Alpha-Beta + 评估** | **对照基线 + 调研** | 完全信息确定性博弈经典解法；但创新分低、已被两份参考方案占满。实现中等强度版本，供"传统 vs 学习"对照实验。 |
| **AlphaZero (PUCT-MCTS+自对弈)** | **主线范式** | Silver et al. 2018(Science) 在将棋(Shogi，与象棋同构)上从零 <24h 超越顶级程序，迁移成熟。唯一痛点：一周样本量不足。 |
| **Gumbel-AlphaZero (Sequential Halving)** | **主线实际训练算法（核心创新）** | Danihelka et al. ICLR 2022。低模拟数(n=16~32)仍**有策略改进的理论保证**，样本效率远超原版(n=800)。这是"一周破局"的算法答案，也是本课程"新算法"的硬抓手。 |
| **MuZero** | **调研/辨析（不实现主线）** | Schrittwieser et al. 2020(Nature)。价值在于环境模型未知；象棋规则完全已知，学 dynamics 是自找麻烦（三网联合、调参爆炸、一周必翻车），且在已知模型博弈上对 AlphaZero 无精度优势。 |
| **DeepCFR / DeepStack / Libratus / Pluribus** | **明确排除 + 辨析** | 全部针对**不完全信息**博弈，求 information set 上的纳什均衡。象棋无隐藏信息，CFR 框架无对象可作用。报告用一段讲清"完全信息 vs 不完全信息"的选型分水岭——这是理解深度的得分点。 |

> **结论**：主线坚定 AlphaZero，**但不复现 2018 原版，而用 Gumbel-AlphaZero**。这既是 SOTA 改进（创新性），又是有限算力/时间下样本效率最优的破局点（可行性）。**选 Gumbel 首先是为吞吐可行性，创新分是副产品**——这个论证逻辑本身写进报告就很有说服力。

## 3. 系统架构（综合系统架构师视角）

### 3.1 架构第一性原则
> **项目成败由"自对弈数据吞吐（局/小时）"单一指标决定。** 整个架构围绕"最大化吞吐"设计，其余决策（网络规模、模拟数、棋盘表示）一律服从。核心矛盾是 **GPU 推理吞吐 ↔ CPU 树搜索速度** 的匹配。

### 3.2 分层模块图

```
L0 规则内核 (C++, perft硬门验证) —— 主线/保底线 100% 共用
   board(mailbox90) · movegen · rules(将军/飞将/重复/和棋) · zobrist · encode
        │（git tag 冻结后只读复用，静态库链接）
        ├─────────────────────────┐
L1-A 保底线(C++)                 L1-B 主线搜索(C++)
   alphabeta+TT+迭代加深+静态搜索    gumbel_mcts(Sequential Halving)+puct
   eval_classic(分阶段手写评估)      虚拟损失 + leaf-batch 队列
        │                              │ 叶子批
        │                  L2 推理服务(C++ LibTorch) 集中式 batch 推理
        │                     收叶子→拼batch(512)→GPU前向→散播(p,v)
        │                     权重热加载(原子指针交换 TorchScript)
        │                              │ 样本(s,π,z) 二进制分片
        │                  L3 ReplayBuffer(Python/共享内存) 环形+近期加权+镜像增广
        │                              │ minibatch
L4 Arena竞技场(C++)            L5 训练循环(Python/PyTorch DDP)
   引擎对弈+BayesElo+gating  ◀──   pretrain(监督)→train_rl→导出权重→通知L2热加载
```

### 3.3 C++/Python 边界 —— 以"权重文件"为唯一契约
- L0 规则、L1 搜索、L2 推理 = **C++**（性能关键）；L3 缓冲、L5 训练 = **Python**（迭代快、DDP 成熟，降一周内最大不确定性）。
- **唯一跨语言契约 = TorchScript `.pt` 权重文件**。不共享内存对象、不 pybind 双向调用。好处：Python 训练崩了，C++ 自对弈/推理仍能用旧权重继续跑——**进程级解耦 = 风险隔离**。
- 自对弈样本用**语言中立二进制分片**（packed 平面 + 稀疏 π + int8 z），Python 侧 mmap 读取。

### 3.4 8 卡分布式拓扑（精修版，吃满 A800）

```
GPU0-1 (2卡) LEARNER : DDP训练(BF16) + Elo-gating；每K步导出candidate→gating→晋升best.pt
GPU2-7 (6卡) INFER+ACTOR : 每卡1个集中式InferServer(常驻best.pt) 服务 ~50个CPU actor线程
共享 ReplayBuffer(2e6局面, mmap, 近期加权) → LEARNER采样
```

**核心杠杆：集中式 batch 推理服务**（不是每个 actor 各持 GPU）。MCTS 单次只产 1 个叶子，单样本前向 GPU 利用率<5%；把几十个 actor 的叶子凑成 batch=512 才能吃满 A800。这是吞吐能否拉满的**唯一架构杠杆**。

| 参数 | 值 | 参数 | 值 |
|---|---|---|---|
| Learner / Infer 卡分配 | 2 / 6 | 推理 batch 目标/上限 | 512 / 1024 |
| leaf-batch 超时窗 | 1~3ms | 虚拟损失 | 3 |
| 每 InferServer 服务 actor | ~50（6×50=300并发局） | 自对弈模拟数(Gumbel) | 32 |
| ReplayBuffer | 2e6 局面 | 训练 batch | 2048 |
| 权重导出周期 K | 1000~2000 步 | Elo-gating | 100局, 胜率>55% 晋升 |

> **弹性纪律**：D4 先用**单卡推理+少量 actor 跑通端到端闭环**（产样本→训练→热加载→Elo 上升），验证正确后再扩到 6 卡。绝不一上来 8 卡——分布式 bug 会淹没算法 bug。

### 3.5 关键 C++ 工程决策表

| 决策点 | 推荐 | 理由 |
|---|---|---|
| 棋盘表示 | **mailbox(90数组)起步** | 正确性优先；蹩马腿/塞象眼用 bitboard 易错；瓶颈在 GPU 不在走法生成。bitboard 列"有余力再优化" |
| 动作编码 | **from→to 8100维+掩码起步**，跑通后切 **2086 紧凑标签集(CCZero)** 做消融 | 8100 零编码 bug；2086 省显存收敛快、社区验证 |
| 推理引擎 | **LibTorch(TorchScript)** | 与训练同框架、热加载简单。TensorRT 提 NPS 但转换踩坑，作 D7 加分项不进关键路径 |
| 推理架构 | **集中式 batch 服务** | 唯一吃满 A800 的架构 |
| 推理抽象 | **`IInferEngine` 接口 + `DummyRandomEngine`** | D1 用 Dummy 跑通整个 MCTS/自对弈管道，**把环境问题与算法问题解耦** |
| 权重热加载 | **原子指针交换 shared_ptr<Module>** | actor 无需停机 |
| 训练框架 | **Python PyTorch DDP** | 迭代快、降风险 |
| 混合精度 | **BF16** | A800 原生，无需 loss scaling |
| 网络规模 | **128/256ch × 10 block 起步** | 小网络自对弈快、batch 大、收敛快；256×19 作消融 |
| 重复/和棋 | **第3次重复判和 + 120回合自然限着**简化版 | 先保收敛，报告注明简化；完整长将/长捉判负列展望 |

## 4. 主线算法设计（综合算法研究员视角）

### 4.1 输入特征平面（C×9×10，C≈14~18 起步）
- 己方7类 + 对方7类棋子 one-hot 平面（14）；当前行动方平面（1，**红黑对称：永远以当前方视角规范化**）；重复计数平面（1~2）。
- **历史帧 T=1 起步**（象棋无气/无提子，历史价值低于围棋，砍历史帧是合理省算力决策，可作消融）。

### 4.2 策略-价值网络
```
Input C×9×10 → Conv3×3(256)+BN+ReLU → ResBlock×N(N=10→20) →
  ├ Policy Head: Conv1×1→FC→2086 logits（+合法掩码，非法置-inf 再 softmax）
  └ Value Head:  Conv1×1→FC(256)→FC(1)→tanh ∈[-1,1]
```
- 加分消融：主干引入 **SE-block / 轻量注意力**（针对车炮长程直线控制），对比纯 ResNet。

### 4.3 损失函数（AlphaZero 标准）
`L = (z−v)² − πᵀlog p + c·‖θ‖²`，c≈1e-4。Gumbel 版 π 用 **completed-Q 改进策略**作目标。

### 4.4 Gumbel-AlphaZero 训练机制（核心）
1. **根节点**：策略 logits 加 Gumbel 噪声 + **Sequential Halving** 在 n 次模拟内从 m 个候选逐轮淘汰，把预算集中给有希望的动作。
2. **非根节点**：基于 completed-Q 的确定性选择，避免 c_puct 高方差调参。
3. **目标策略**：completed-Q 构造改进策略，**小 n 也保证 policy improvement**。
- 对照用原版 PUCT：`a*=argmax[Q+c_puct·P·√ΣN/(1+N)]` + 根节点 Dirichlet(α=0.2, ε=0.25)。

### 4.5 一周破局三招（缺一不可）
1. **监督预训练冷启动**：公开人类棋谱（东萍/PGN-CN，数十万~百万局）监督学 policy(预测人类走子)+value(预测胜负)，1~2 天达业余强手，**跳过随机乱走的痛苦早期**（AlphaGo 2016 两阶段思想）。
2. **Gumbel 低模拟自对弈**：n 从 800 降到 32，**同 GPU 时间产出约 25 倍对局**。
3. **8 卡 actor-learner 分布式**：把算力转成样本量。
> 闭环：冷启动解决"起点低"，Gumbel 解决"单局贵"，分布式解决"吞吐"。

## 6. 实验与评估体系 + 实验亮点（综合测试策略师视角，并回答"亮点在哪"）

### 6.1 正确性验证（最高优先级）
- **perft 验证**：`perft(d)` 叶子计数 + `perft_divide` 定位 bug；无权威数值则**双实现交叉对拍**（naive 慢生成器 vs 主生成器，逐位一致）。起始/中局/残局/将军受限局面，d=1~5。
- **规则单元测试**：将军/应将/飞将/马腿象眼炮架/兵卒过河/自杀着过滤/长将判负/重复判和/困毙。**报告须声明采用的长将规则版本**，且自对弈/AB/Pikafish 三者规则一致（否则系统性偏差）。
- **一致性**：Zobrist make/unmake 对称、FEN 往返、**网络输入编码↔解码无损往返**（防"喂给网络的局面≠真实局面"隐形 bug）。

### 6.2 棋力评估（统计站得住脚）
- **BayesElo / Ordo** 反推 Elo（处理平局、给 ±95% CI），不用在线增量公式。
- **配对开局 + 固定开局簿 + 红黑各半**消除先手偏置与重复对局。
- **局数**：消融统一 **400 局起步**（分辨~70 Elo 差），关键结论加到 800~1000；可用 **SPRT** 提前停止省算力。
- **Pikafish 天梯锚定**：限制其 nodes/depth 调档，找本方胜率≈50% 的档位 → 绝对棋力锚点。

### 6.3 效率指标
NPS（MCTS模拟/秒、AB节点/秒）、每步模拟数↔耗时曲线、GPU利用率、**自对弈吞吐(局/小时)**、推理延迟、置换表命中率、剪枝率/EBF、**多卡扩展效率(1/2/4/8卡)**、峰值显存。

### 6.4 消融实验矩阵（坐实创新性）
统一：单变量、配对开局、400局、BayesElo+CI。

| 实验组 | 变量 | 目的/亮点 |
|---|---|---|
| **1 模拟数 n_sim** | {16,32,64,200,800} | 证明 Gumbel 低模拟即强，找棋力-算力拐点 |
| **2 冷启动有无** | 纯自对弈 vs 预训练+自对弈 | 量化样本效率，1周约束下最关键 |
| **3 网络规模** | 6/10/20 block | 效率-棋力性价比 |
| **4 输入历史帧** | T=1/2/4/8 | 象棋是否需要历史帧 |
| **5 ★旗舰对照** | AlphaZero vs Gumbel-AZ **vs Alpha-Beta（同时间预算 100ms/500ms/1s）** | 学习型 vs 经典搜索的 crossover 点——**报告最大亮点** |
| 6 选做 | SE-block/注意力、损失权重、Dirichlet | 网络结构探索 |

### 6.5 训练监控（防静默坍缩）
- 训练侧：total/policy/value loss、value 符号准确率、**policy 熵**（骤降=坍缩，不降=没收敛）、梯度范数。
- 棋力侧（最终裁判）：对历史版本胜率(>55% gating)、对固定 Pikafish 档胜率曲线、**对局长度分布**（过短=乱杀坍缩）、平局率。
- 固定锚点：每 N 个 checkpoint 跑 100 局快速 Elo，画 **Elo-vs-训练步曲线**。
- 诊断局面集：经典"两步杀/明显得子"局面，测 policy top-1 命中（可解释定性进度）。

## 7. ★实验亮点 / 新技术新算法（直接回答你的两个问题）

### 7.1 我们用到的"比较新 / 新算法"

| 技术 | 新度 | 在本项目的角色 | 为什么是亮点 |
|---|---|---|---|
| **Gumbel-AlphaZero（Sequential Halving 规划策略改进）** | **ICLR 2022，2018 原版 AlphaZero 之后的 SOTA 改进** | **主线训练算法** | 这是本项目的"新算法"硬核：低模拟数(16~32)仍有**策略改进理论保证**，样本效率比原版高一个数量级。班上做纯 Alpha-Beta(1990s 技术) 的同学完全没有"新算法"可言——这一项就拉开代差。有 DeepMind 论文背书，老师可查证。 |
| **监督冷启动 + RL 微调两阶段管线** | 工程范式创新 | 破局一周约束 | 把"算力受限下的破局"上升为**可量化的方法贡献**（消融出"提速 N 倍"），不是简单复现。 |
| **8 卡分布式 actor-learner + 集中式 batch 推理** | 现代大规模 RL 工程标配 | 吞吐引擎 | 单机方案物理上做不出的**多卡扩展效率曲线 + 自对弈吞吐热力图**，直接吃满"效率"分。 |
| **领域定制：2086 紧凑动作空间 + 合法掩码 + 红黑对称规范化 + 砍历史帧** | 针对象棋棋种的适配 | 输入/动作设计 | 体现"不是无脑套围棋，而是理解象棋几何/规则做定制"。 |
| SE-block/轻量注意力增强主干（选做） | CVPR 2018 | 消融加分 | 针对车炮长程控制的结构探索。 |

> **一句话**：新算法 = **Gumbel-AlphaZero（2022）**；新范式 = **监督冷启动 + 低模拟自对弈 + 8 卡分布式**。对比"任务书点名但已是经典"的 Alpha-Beta(基线) / MuZero·DeepCFR(辨析不用)，我们把"新"用在了刀刃上，且每一项"新"都对应任务书允许的技术族（MCTS 系），不偏离要求。

### 7.2 实验要突出什么（三张"杀手锏图"，别人物理上做不出）

1. **🔥 训练步数 → Elo 上升曲线**（最核心一张图）
   - 纯 Alpha-Beta 方案**没有"训练"概念，画不出任何学习曲线**。这张图本身宣告代差。
   - 加强版：同图叠加"纯自对弈" vs "预训练+自对弈"两条线，一图可视化样本效率创新（对应消融2）。

2. **🔥 AlphaZero/Gumbel-AZ vs Alpha-Beta 同时间预算对照**（对应消融5）
   - 固定每步 100ms/500ms/1s，比 Elo、决策质量。**展示 crossover 点**：短预算 AB 占优、长预算学习型反超。这是两份纯传统方案给不出、最有说服力的"研究结论"。

3. **🔥 Gumbel n_sim 消融曲线（16→800）**（对应消融1）
   - 证明"用 16~32 模拟达到原版数百模拟的效果"。把"算法创新"翻译成铁证，且有 ICLR 2022 出处。

辅助亮点图：8 卡 GPU 利用率/自对弈吞吐热力图、引擎自我进化天梯（v1→vN Elo 阶梯）、多卡扩展效率曲线。

### 7.3 把项目讲成"研究"而非"作业"（报告 thesis）
> **在仅 8 卡 A800、一周训练预算的强约束下，通过"人类棋谱监督预训练冷启动 + Gumbel-AlphaZero 低模拟自对弈"两阶段管线，可在中国象棋上训练出显著超越同等时间预算 Alpha-Beta 引擎的学习型博弈 AI；并通过系统性消融与对照实验，量化刻画样本效率、模拟预算与最终棋力之间的权衡关系。**

这句话同时命中：研究问题（有限预算下学习型能否赢）、方法（两阶段+Gumbel）、验证（消融+对照）、贡献（量化权衡）。**每个创新点配一张图，图题写成结论句**（"图5：预训练使 Elo 收敛速度提升约 N 倍"），让老师不读正文也能 get 贡献。
